# 分散クラウドと単一ノード クラウドの比較

Rejourney は、次の 2 つの公式セルフホスト展開形式をサポートしています。

- 1 つのサーバーの場合は **シングルノード Docker Compose**、または VPS
- 実稼働クラスターおよび水平スケーリング用の **配布済み K3s**

どちらも同じコア バックエンド モデルを使用するようになりました。

- ストレージ エンドポイントはデータベースに支えられています
- 取り込みアップロードはバックエンド所有のアップロードリレーを経由します
- ワーカーが検証済みのアーティファクトを処理する
- リプレイの可視性はアーティファクト主導型です

---

## 機能の比較

|特集 |分散型クラウド |単一ノードのクラウド |
|---------|--------------------|-------------------|
|プラットフォーム | K3s | Docker Compose |
|スケール |マルチノード |単一ノード |
|パブリック エントリポイント | Traefik 入口 | Traefik コンテナ |
|アップロードパス | API + インジェストアップロード サービス | API + インジェストアップロード サービス |
|ストレージの信頼できる情報源 | `storage_endpoints` テーブル | `storage_endpoints` テーブル |
|デフォルトのオブジェクトストレージ |外部 S3 |内蔵 MinIO |
|外部 S3 のサポート |はい |はい |
|秘密の暗号化 | `STORAGE_ENCRYPTION_KEY` | `STORAGE_ENCRYPTION_KEY` |
|アップデートの流れ | k8s デプロイ + ジョブ | `deploy.sh update` |

---

## 共有ストレージモデル

どちらの展開モデルでも、ランタイム ストレージ構成は、env フォールバックではなく、Postgres から取得されます。

つまり:

- アクティブなオブジェクト ストレージ エンドポイントは `storage_endpoints` に保存されます
- シークレット アクセス キーは `key_ref` に暗号化されます
- ランタイムはデータベース行を読み取ります
- ブートストラップ/インストール スクリプトは、`.env` 入力をデータベース行に同期する役割を果たします。

これにより、セルフホスト型 Docker は、古いフォールバック モデルよりも本番および local-k8s にかなり近づきます。

---

## 単一ノード Docker Compose を選択する場合

次の場合は、Docker Compose を選択してください。

- 1 つの VPS またはベアメタル ホストにデプロイしている場合
- 最速のインストール パスが必要な場合
- デフォルトで組み込みの MinIO が必要
- マルチノードのスケーリングや Kubernetes 操作は必要ありません

公式エントリーポイント:

- `docker-compose.selfhosted.yml`
- `scripts/selfhosted/deploy.sh`
- `docs/selfhosted/README.md`

---

## 分散型 K3s を選択する場合

次の場合は、K3s を選択してください。

- 複数のノードが必要です
- Kubernetes ネイティブの操作とシークレット処理が必要
- API、アップロード、ワーカー サービスを個別にスケーリングしたい場合
- ローリング デプロイとより強力なインフラ分離が必要な場合

K3s パスは、`k8s/` および `scripts/k8s/` の下に存在します。

---

## 操作上の違い

主な違いはデータ モデルではなくなりました。動作形状です:

- Compose: 1 つのマシン、1 つの Docker ネットワーク、1 つのオペレーター スクリプト
- K3s: 複数のポッド、名前空間、クラスターイングレス、Kubernetes ジョブおよびシークレット

---

## 実践的な指導

すぐに自己ホストしたい場合は、単一ノード Compose から始めてください。

必要な場合は、K3s に移動してください。

- より多くのスループット
- ローリングクラスターデプロイ
- 水平スケーリング
- より回復力の高いインフラストラクチャの分離
