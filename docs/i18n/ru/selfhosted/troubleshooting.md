# Самостоятельное устранение неполадок

Используйте эту страницу, если вы подписались на [Self-hosted Rejourney](/docs/selfhosted) и что-то не работает или ведет себя странно. Команды выполняются из **корень репозитория** (где находится `docker-compose.selfhosted.yml`).

---

## Быстрые проверки

### Статус услуги

```bash
./scripts/selfhosted/deploy.sh status
```

### Журналы API

```bash
./scripts/selfhosted/deploy.sh logs api
```

### Загрузить журналы ретрансляции

```bash
./scripts/selfhosted/deploy.sh logs ingest-upload
```

### Рабочие журналы

```bash
./scripts/selfhosted/deploy.sh logs ingest-worker
./scripts/selfhosted/deploy.sh logs retention-worker
./scripts/selfhosted/deploy.sh logs alert-worker
```

## 1. Сбой установки или обновления до или во время начальной загрузки.

### Симптомы

- `bootstrap` выходит из ненулевого значения
- сервисы приложений никогда не станут здоровыми
- `status` показывает API или рабочие процессы, ожидающие начальной загрузки.
- установить или обновить выходы с помощью `Database authentication failed before bootstrap.`

### Чеки

```bash
docker compose -f docker-compose.selfhosted.yml --env-file .env.selfhosted logs bootstrap
```

Распространенные причины:

- плохо `DATABASE_URL`
- несоответствие учетных данных (например, из-за предыдущего неудачного развертывания)
- отсутствует `STORAGE_ENCRYPTION_KEY`
- недействительные учетные данные S3
- неверный URL-адрес внешней конечной точки S3
- на **ARM64** отсутствует поддержка изображений (установите `DOCKER_DEFAULT_PLATFORM=linux/amd64` или используйте `./scripts/selfhosted/deploy.sh`, который устанавливает ее, если она не установлена)

Восстановление:

1. Если у вас остался оригинальный `.env.selfhosted`, восстановите его и запустите:

```bash
./scripts/selfhosted/deploy.sh update
```

2. Если вам не нужны старые данные, сотрите и установите заново:

```bash
./scripts/selfhosted/deploy.sh reset
./scripts/selfhosted/deploy.sh install
```

**Сообщения схемы/миграции:** При обычной установке база данных запускается пустой, и программа начальной загрузки все устанавливает. Если вы используете **восстановил Postgres из резервной копии** на новом сервере, но метаданные миграции отсутствуют или вы указали стек на **неправильная база данных**, загрузочная программа может завершиться с ошибкой о несогласованной базе данных вместо того, чтобы перезаписать ваши данные. Если вы не выполняете расширенное восстановление, исправьте `DATABASE_URL` и восстановите согласованную резервную копию или начните с чистого тома. Для преднамеренного восстановления только для миграции в некоторых установках используется `REJOURNEY_ALLOW_ORPHAN_DB_MIGRATE_ONLY=1` в `.env.selfhosted` (перед использованием см. документацию сопровождающего или службу поддержки).

### Исправить

1. Если у вас есть оригинальный `.env.selfhosted`, восстановите его и запустите заново:

```bash
./scripts/selfhosted/deploy.sh update
```

2. Если у вас нет оригинального `.env.selfhosted`, вайпните и установите заново:

```bash
./scripts/selfhosted/deploy.sh reset
./scripts/selfhosted/deploy.sh install
```

`update` повторно запускает синхронизацию схемы, начального числа и конечной точки хранилища. `reset` удаляет локальные контейнеры и тома данных, поэтому новая установка может безопасно генерировать новые учетные данные.

## 2. Сеансы засчитываются, но повтор остается пустым.

### Что это обычно означает сейчас

При нынешней архитектуре это обычно одно из двух:

- `ingest-upload` не удалось сохранить байты артефакта.
- `ingest-worker` не удалось обработать загруженный артефакт.

Устройство больше не загружает данные напрямую в MinIO/S3, поэтому доступность корзины с телефона больше не является главным подозреваемым.

### Чеки

```bash
./scripts/selfhosted/deploy.sh logs ingest-upload
./scripts/selfhosted/deploy.sh logs ingest-worker
./scripts/selfhosted/deploy.sh logs api
```

Искать:

- `artifact.upload_received`
- `artifact.upload_stored`
- `artifact.retry`
- `artifact.failed`
- `session.reconciled`
- `session.finalized`

### Распространенные причины

- неправильные учетные данные S3 в `.env.selfhosted`
- отсутствует внешний ковш S3
- внешняя конечная точка S3 недоступна из сети Docker
- загрузка реле неработоспособна
- работник застрял при повторной попытке неудачных артефактов

### Исправить

- проверьте значения `S3_*`
- если вы изменили конфигурацию хранилища, повторите:

```bash
./scripts/selfhosted/deploy.sh update
```

## 3. Панель мониторинга загружается, но вызовы аутентификации или API завершаются неудачно.

### Чеки

- Хост панели мониторинга DNS указывает на сервер
- Хост API DNS указывает на сервер
- принимающий хост DNS указывает на сервер
- порты `80` и `443` открыты
- Let’s Encrypt выдал сертификаты

Осмотрите:

```bash
./scripts/selfhosted/deploy.sh logs traefik
./scripts/selfhosted/deploy.sh logs api
```

---

## 4. TLS или проблемы с сертификатом.

Traefik автоматически управляет сертификатами.

### Чеки

```bash
dig example.com
dig api.example.com
dig ingest.example.com
dig www.example.com
```

Убедитесь, что оба имени соответствуют серверу, на котором работает стек.

Если DNS был неправильным во время первой установки, исправьте DNS и перезапустите:

```bash
./scripts/selfhosted/deploy.sh update
```

---

## 5. Внешний S3 работает в CLI, но Rejourney не может загрузить

Помните, что путь загрузки — серверный.

Важный сетевой путь:

- Контейнер `ingest-upload` -> ваша конечная точка S3.

Протестируйте с сервера, просмотрев журналы ретрансляции и подтвердив конечную точку/сеть/ключи в `.env.selfhosted`.

Если вы их изменили, повторите:

```bash
./scripts/selfhosted/deploy.sh update
```

---

## 6. Встроенная установка MinIO, но артефакты все равно не работают

### Чеки

```bash
./scripts/selfhosted/deploy.sh logs minio
./scripts/selfhosted/deploy.sh logs minio-setup
```

Одноразовый запрос `minio-setup` должен создать сегмент с именем `S3_BUCKET`.

Если вы изменили имя корзины после первой установки, запустите:

```bash
./scripts/selfhosted/deploy.sh update
```

## 7. На страницах биллинга отображается отключенный биллинг.

Это ожидается, если не настроены ключи Stripe.

Стек больше не отключает выставление счетов, поскольку он «самостоятельно размещен». Он отключает выставление счетов, поскольку Stripe не настроен.

Если вы не установили ключи Stripe:

- Пользовательский интерфейс выставления счетов остается в автономном/неограниченном состоянии
- Stripe оформление заказа и веб-перехватчики остаются отключенными

---

## 8. Конечная точка хранилища в Postgres неверна после изменения `.env.selfhosted`.

Бегать:

```bash
./scripts/selfhosted/deploy.sh update
```

Путь обновления повторно запускает загрузку и повторно синхронизирует активную строку `storage_endpoints`.

---

## 9. Необходимость остановки служб без потери данных

Использовать:

```bash
./scripts/selfhosted/deploy.sh stop
```

Это останавливает только контейнеры. Тома не удаляются.
